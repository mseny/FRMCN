1 De la terminal al botón

Haz recuento de lo que ya tienes: un vault que guarda el conocimiento, un vectorizador que lo busca por significado, un tejedor que lo conecta solo y un equipo de agentes que lo mantiene sano. Un cerebro de empresa completo. Pero tiene un problema de puerta: para usarlo hay que abrir Obsidian o teclear en Claude Code. Para ti, perfecto. Para tu socio, tu comercial o tu diseñadora… una barrera. El conocimiento existe, pero solo lo aprovecha quien sabe entrar.

La solución de hoy: un portal interno — una pequeña web app, hecha a medida y solo para tu equipo, que se sienta encima del cerebro y lo convierte en botones. Un panel que responde "¿cómo va todo?" de un vistazo, un tablero para mover tareas y una puerta por donde entra y sale la información. Es el mismo patrón que usan estudios y agencias para su "cockpit" interno, en versión esencial.

Analogía — el barco: el cerebro que montaste en las clases anteriores es la sala de máquinas: potente, pero ahí abajo solo entra el ingeniero. El portal es el puente de mando: indicadores a la vista y palancas que cualquiera de la tripulación sabe accionar. Mismo barco, misma maquinaria — lo que cambia es quién puede gobernarlo.

El giro de esta clase: no programar, sino saber nombrar

Igual que en fotografía no hacía falta saber de cámaras sino de vocabulario (¿recuerdas las 7 capas?), para encargarle software a Claude no hace falta saber programar: hace falta saber nombrar las piezas. Quien dice "hazme una web para ver cómo va mi negocio" recibe una adivinanza; quien dice "una navbar, tres KPI cards, un gráfico de líneas y una tabla con badges de estado" recibe exactamente su plano. Por eso la primera mitad de la clase es un diccionario y la segunda, las tres pantallas del portal.

2 Diccionario (I): la estructura de una web

Toda web —tu portal, tu banco, tu periódico— se construye con el mismo esqueleto. Aprenderse estas cinco o seis palabras vale para encargar cualquier software, no solo el de hoy. Empecemos por el plano general:

Analogía — un piso: una web se describe como una vivienda. La navbar es el recibidor (te orienta nada más entrar), la sidebar es el pasillo con las puertas de cada habitación, el área principal es el salón donde pasa la vida, y el footer es el trastero: ahí van las cosas legales que nadie mira pero tienen que estar.

El esqueleto universal: con estas cuatro zonas se describe casi cualquier aplicación.

Navbar

La barra superior fija: logo, secciones principales y el usuario. Es el mapa del portal — desde cualquier pantalla te dice dónde estás y a dónde puedes ir.

Sidebar

El menú lateral, siempre a la vista. Se pide cuando hay muchas secciones; si tu portal tiene tres pantallas, con la navbar basta. Menos es más.

Área principal

Donde vive el contenido de cada pantalla. Todo lo demás (navbar, sidebar, footer) se repite; esto es lo que cambia cuando navegas.

Grid de tarjetas

La retícula que ordena la información en filas y columnas de cards (tarjetas). "Un grid de 3 columnas con una card por proyecto" es una frase que Claude entiende a la primera.

Vista / ruta

Cada "pantalla" de una app es una vista con su dirección propia (/panel, /clientes). Describe tu software vista a vista: es como Claude lo va a construir.

Responsive

Que se adapte a móvil, tablet y escritorio. Pídelo siempre de serie: "y que sea responsive". Tu portal se consultará más desde el sofá que desde la oficina.

Regla de oro del encargo: describe tu software pantalla a pantalla (las vistas) y cada pantalla pieza a pieza (los componentes de la siguiente sección). La estructura son las habitaciones; los componentes, los muebles. Con las dos listas, Claude no adivina: ejecuta.

4 El motor: Claude por API

Aquí está la novedad técnica de la clase. Hasta ahora usabas Claude conversando — en la web de Claude o en Claude Code. Tu portal necesita otra cosa: que Claude responda cuando un programa se lo pide, sin que haya nadie chateando. Eso es la API (interfaz de programación): el enchufe de Claude para software. Tu portal se conecta a ese enchufe cada vez que alguien pulsa un accionable o pregunta algo al cerebro.

Claude Code · el que CONSTRUYE

El arquitecto. Lo usas tú, en la terminal, para crear el portal y para cambiarlo ("añade una columna al kanban"). Trabaja una vez y se va. Es el mismo Claude Code de todo el módulo — y trabaja con tu suscripción: eres tú, con tu cuenta, tecleando en tu máquina.

Claude por API · el que VIVE DENTRO

El mayordomo. Vive dentro del portal y atiende cada timbrazo: un botón pulsado, una pregunta en el chat, un archivo en la ingesta. Trabaja cada vez que alguien lo llama, para cualquiera del equipo — y con contador propio: la API key, facturada por uso.

Este patrón —recuperar primero en tu base, generar después— es el corazón de casi todo el software con IA. Si quieres impresionar en una reunión: se llama RAG (Retrieval-Augmented Generation).

"Pero yo ya pago Claude…" — suscripción vs. API

Aviso importante para no llevarse sorpresas: tu suscripción de Claude (el plan Team con el que chatea tu equipo) y la API son dos productos distintos con dos facturas distintas.

Suscripción (Team)

Paga a las personas: cada miembro usa Claude por una cuota fija al mes — el chat de claude.ai y su propio Claude Code en la terminal (por eso construir el portal no genera factura de API). Tu portal no puede usarla: no hay una persona trabajando, hay un programa atendiendo a otros.

API (Console)

Paga al software: se contrata en la consola de desarrollador de Anthropic, tiene su propia API key y se factura por uso (por texto procesado). Un portal interno de equipo suele costar poco — pero es otra factura, y conviene ponerle un límite de gasto mensual desde la consola.

"¿Y por qué no puede el portal usar mi suscripción?"

Es la pregunta lógica: si mi plan ya incluye Claude Code, ¿por qué pagar API? Porque suscripción y API responden a dos situaciones distintas, y la frontera son tres limitaciones técnicas (y de licencia) que conviene conocer:

La butaca es personal

Tu suscripción autentica a una persona: tú, con tu login, en tu sesión. El portal es un programa que atiende a todo el equipo — software sirviendo a terceros. Eso es, por definición y por licencia, territorio de la API.

Los límites son de persona

Los planes tienen ventanas de uso pensadas para una persona trabajando: si te pasas, toca esperar. Un portal con varios usuarios (o con tareas programadas) las agotaría y se quedaría colgado a mitad de informe. La API factura por consumo y no se corta — y el tope de gasto lo pones tú.

La credencial es distinta

Tu login es interactivo: navegador, contraseña, tu sesión en tu máquina. Un servidor necesita una llave que funcione desatendida, viva en el .env y se pueda revocar sin cambiarte la contraseña. Eso es exactamente la API key.

Regla del pulgar: pregúntate quién está usando a Claude. Si eres tú en tu máquina (construir el portal, mantenerlo, tus consultas en la terminal) → tu suscripción, que ya la pagas. Si es tu software trabajando para otros o desatendido (el equipo pulsando accionables, tareas programadas de madrugada) → API. Los dos modos conviven en el mismo proyecto: el arquitecto trabaja con tu butaca; el mayordomo necesita contador propio.

Analogía — el contador de la luz: la API key es la tarjeta de tu contador eléctrico: identifica quién consume y a quién le facturan. Por eso es una credencial secreta: si la dejas a la vista, cualquiera enchufa sus aparatos a tu contador. Vive en un archivo .env en el servidor, nunca en el código que ve el navegador, nunca en el vault, nunca en un repositorio público.

Privado ≠ local: el matiz honesto

El mantra del módulo era "todo en local, nada sale del equipo". El portal lo matiza: cuando usas Claude por API, la pregunta y el contexto que le adjuntas sí viajan a los servidores de Anthropic. ¿Deja de ser privado? No — pero hay que decirlo con precisión:

Local

No lo ve nadie. El vault, el vectorizador y el tejedor siguen aquí: nunca salen de tu máquina.

API de Anthropic

Lo procesa Anthropic bajo contrato comercial: confidencial, no entrena modelos con tus datos y con retención acotada. Privado, pero con un tercero de confianza en medio.

Público

Lo que subes a redes, webs abiertas o herramientas gratuitas sin contrato. Eso es lo contrario de privado.

Las tres reglas del portal: (1) los datos personales sensibles siguen fuera del vault — la norma de siempre, y ahora con doble motivo: lo que no está en el cerebro no puede viajar a ninguna API; (2) tú decides qué contexto viaja — solo las notas relevantes que elige el vectorizador, nunca el vault entero; (3) la API key es una credencial — al .env del servidor y a ningún sitio más. Frase para la pizarra: local = no lo ve nadie; API = lo ve Anthropic bajo contrato, no lo entrena, no lo publica.

5 Pantalla 1 — El panel: KPIs, objetivos y accionables

La landing de tu portal — lo primero que ve quien entra — tiene un único trabajo: responder "¿cómo va todo?" en cinco segundos. Tres franjas, de arriba abajo: los números (KPI cards), el estado (objetivos y tareas con sus badges) y las palancas (los accionables) — más el chat abajo del todo, para preguntarle al cerebro en lenguaje natural. El detalle vive en las otras pantallas.

Los accionables: botones que trabajan

La pieza estrella del panel. Un accionable no navega a otra pantalla: ejecuta una tarea completa, de principio a fin. Cada accionable empaqueta detrás de un botón algo que hasta hoy hacías tú a mano en la terminal: leer del cerebro (el vault y los originales archivados en procesados/) → producir un resultado con Claude → dejarlo en su sitio: respuesta en el chat o documento en salidas/. Para definir uno bien, rellena siempre estas cuatro casillas:

Accionable Lee… Produce… Lo deja en…
⚡ Informe semanal Las notas de los últimos 7 días Un resumen ejecutivo Documento en salidas/ + pantalla
⚡ Propuesta de post Las novedades del cerebro Un borrador para redes salidas/, para revisar y copiar
⚡ Resumen de reunión La última acta de procesados/ Acuerdos + tareas detectadas Nota + tarjetas en el tablero

Analogía — las palancas del puente: en el puente de mando no hay un cable por cada máquina: hay palancas con la maniobra completa detrás. "Informe semanal" es una palanca: quien la acciona no necesita saber que por debajo se busca en el vault, se llama a la API y se guarda una nota. Esa es la gracia: tu conocimiento técnico, empaquetado para que lo use cualquiera.

6 Pantalla 2 — El tablero: un kanban

El panel te dice cuánto hay; el kanban te deja moverlo. Es la pieza de gestión visual más rentable que existe: columnas que son estados, tarjetas que son tareas, y arrastrar que es avanzar. Nuestro consejo: empieza con un tablero para un solo proyecto — el activo. El multi-proyecto llega cuando el simple se quede corto, no antes.

📋 Por hacer

Lo comprometido que aún no ha empezado.

⚙️ En curso

En lo que se está trabajando ahora. Pocas tarjetas: si hay diez, no hay foco.

👁️ Validar

Trabajo terminado que espera tu ojo antes de darse por bueno. Imprescindible cuando parte del trabajo lo hace la IA.

✅ Hecho

Validado y cerrado. De aquí se alimenta el informe semanal.

Fíjate en la columna "Validar": no es adorno. En un negocio donde los agentes redactan, resumen e ingieren por ti, necesitas un lugar donde el trabajo hecho por la máquina espere la aprobación de la persona. Es la misma regla de todo el módulo (proponer antes de escribir), convertida en columna.

La regla que lo cambia todo: la fuente de verdad

Aquí viene la decisión de arquitectura más importante del portal, y vale para todas sus pantallas: el portal no guarda nada importante por su cuenta. Cada tarea del kanban vive como nota en el vault; el tablero es solo una vista. ¿Arrastras una tarjeta a "Hecho"? Se actualiza su nota. ¿Editas la nota a mano en Obsidian? El tablero lo refleja. Un solo dueño de los datos —el cerebro— y todas las herramientas mirándolo. Así jamás tendrás dos versiones de la verdad.

7 Pantalla 3 — Ingesta y salida: cerrar el bucle

Hasta aquí, el portal lee el cerebro. Falta el bucle completo: que también lo alimente (ingesta) y que los resultados puedan salir hacia el mundo (salida). La ingesta acepta lo que de verdad circula por un negocio — PDF, Word, Excel y PowerPoint, además de texto — y el ciclo de vida tiene tres paradas: la bandeja de entrada (lo que sueltas), procesados/ (los originales ya convertidos en notas, archivados con su fecha) y salidas/ (los documentos que el portal genera). El portal se convierte así en la membrana del cerebro: todo lo que entra pasa por ella, todo lo que sale lleva su sello.

ENTRA · la ingesta

Una zona de arrastrar y soltar para PDF, Word, Excel y PowerPoint (y .md/.txt), más una caja para pegar texto. Claude analiza el contenido y redacta notas según las convenciones de tu vault: frontmatter, etiquetas y enlaces.

EL CEREBRO

La nota aterriza en su carpeta, el enlazador la conecta y el tejedor vigila que no quede huérfana. El original se archiva en procesados/ con su fecha: queda constancia, y los accionables pueden releerlo.

SALE · los resultados

Los accionables leen el vault y procesados/, y responden de dos formas: en el chat (respuestas al momento) o generando documentos que aterrizan en salidas/ — visibles en pantalla y descargables.

Analogía — la aduana: en una aduana nada cruza sin registrarse: lo que entra se inspecciona y se etiqueta; lo que sale lleva su documentación. La ingesta es exactamente eso — la vista previa es el inspector (Claude te enseña las notas que propone crear y no escribe nada hasta que tú apruebas) y procesados/ es el registro: el original queda archivado y sellado con su fecha. La regla de edición segura del módulo, aplicada a la puerta de entrada.

Prompt · ingesta y salida
Cierra el circuito de mi portal con la pantalla "Ingesta". STACK: el mismo del portal (Next.js + Tailwind; Claude desde el servidor con @anthropic-ai/sdk y el modelo claude-opus-4-8). Al pulsar "Ingerir": extrae el texto de cada archivo y pídele a Claude que lo convierta en una o varias notas SEGÚN LAS CONVENCIONES DE MI VAULT: frontmatter (título, tipo, etiquetas area/... y proyecto/...), texto limpio en markdown y enlaces [[...]] a las notas existentes que mencione. Pásale a Claude una nota real de mi vault como ejemplo, para que copie el formato exacto. 3. OBLIGATORIO: vista previa de las notas propuestas; no escribas nada en el vault hasta que yo apruebe. 4. Estado de carga durante el procesado, toast al terminar y el estilo del resto del portal.

En una frase

El portal es la puerta bonita del cerebro: el panel lo muestra, el kanban lo mueve y la ingesta lo alimenta. Lo que entra se hace nota (y su original queda en procesados/); lo que se genera aterriza en salidas/. Por debajo no hay magia nueva — hay tu vault, tu vectorizador, tus agentes y Claude por API, empaquetados detrás de botones que cualquiera de tu equipo sabe pulsar. Y las reglas de siempre mandan: la fuente de verdad es el cerebro, nada se escribe sin aprobar, y el PII sensible ni entra ni viaja.
