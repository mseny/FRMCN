1 El Ecosistema Completo

Qué puede crear y configurar un usuario dentro del entorno operativo de Cowork. La jerarquía de complejidad va desde una simple instrucción (Skill) hasta la conexión con aplicaciones externas (Connector).

SKILL.md

Instrucciones especializadas que Claude carga bajo demanda. El "manual de trabajo" para una tarea concreta.

AGENT.md

Subagente con su propio sistema de instrucciones, modelo y herramientas. Una instancia Claude separada con personalidad propia.

Hooks

Automatizaciones que se disparan en eventos del sistema. El mecanismo real de "workflows" automáticos en Cowork.

Plugin (.plugin)

Paquete instalable que agrupa Skills + Agents + Hooks + MCP. Un módulo que amplía Cowork con nuevas capacidades.

Connector (MCP)

Conexión a una herramienta externa (Slack, Notion, HubSpot…). Da a Claude acceso real a tus datos y apps.

Project / Workspace

Carpeta local montada en Cowork. Claude puede leer y escribir todos los archivos que contiene de forma nativa.

Scheduled Task

Tarea que corre automáticamente según un calendario (cron). Claude ejecuta un prompt sin intervención del usuario.

Artifact

Página HTML persistente con datos en vivo de conectores. Un dashboard que el usuario puede reabrir cuando quiera.

Memory (CLAUDE.md)

Archivo de memoria persistente entre sesiones. Almacena preferencias, contexto de equipo y hechos duraderos.

La jerarquía de complejidad: Skill (instrucciones) → Agent (instancia propia) → Plugin (bundle de todo) → Connector (acceso a apps externas) → Scheduled Task (automatización temporal).

2 SKILL.md — Conocimiento bajo demanda

Una Skill es un archivo Markdown que le dice a Claude cómo hacer algo específico. Claude lo lee automáticamente cuando detecta que la petición del usuario encaja con la descripción del skill, sin que el usuario tenga que invocarlo explícitamente.

Analogía: Un skill es como el manual de procedimientos de un departamento. Cuando Claude detecta que tiene que hacer algo del ámbito del skill, saca el manual del cajón y lo consulta antes de actuar.

Anatomía del Archivo
--- ← PARTE 1: FRONTMATTER YAML (Puerta de entrada) --- name: nombre-del-skill description: "Cuándo activar este skill. Claude lee este texto para decidir si el skill aplica. Aquí van los TRIGGERS: 'analiza mis campañas', 'qué ROAS tengo', 'dame insights'. También actívalo cuando el archivo tenga columnas de CTR..." --- # PARTE 2: CUERPO MARKDOWN (Instrucciones reales) ## Qué hacer cuando el skill se activa Instrucciones paso a paso para Claude... ## Código de referencia ```python # Snippets que Claude usará directamente ``` ## Output obligatorio Qué generar, en qué formato, qué archivos crear...

Los 3 niveles de carga (Progressive Disclosure)

Esta es la clave para entender cómo afectan los skills al consumo de tokens de la ventana de contexto:

1
Metadata (Siempre en contexto)

Solo name + description (~100 palabras). Costo mínimo. Claude siempre lo tiene disponible para decidir si activar.

2
SKILL.md Body (Solo al activarse)

Las instrucciones detalladas (<5.000 palabras). Solo consumen tokens cuando Claude decide que el skill aplica a la tarea actual.

3
references/ (Bajo demanda)

Docs adicionales o scripts. Claude los lee solo si los necesita. Nunca ocupan contexto innecesariamente.

Estructura de carpetas completa

skill-name/
├── SKILL.md ← obligatorio
├── references/ ← documentación (bajo demanda)
│ ├── guia-avanzada.md
│ └── casos-de-uso.md
├── scripts/ ← código reutilizable (ejecutable)
│ └── procesar.py
└── examples/ ← ejemplos de input/output

3 AGENT.md y Soul (Personalidad)

Un Agent es más que un skill: es una instancia de Claude separada con su propia ventana de contexto, su propio system prompt y acceso restringido a herramientas específicas.

Skill vs Agent — La diferencia clave:

Un Skill inyecta instrucciones en el Claude principal (mismo contexto).
Un Agent es una instancia Claude nueva con su propio contexto. Consume más tokens pero tiene más independencia.

Anatomía de AGENT.md
--- name: nombre-del-agente # kebab-case, 3-50 chars description: "Cuándo activar este agente. Debe incluir bloques <example> con contexto + diálogo." <example> Context: El usuario tiene datos de ventas del mes user: "Analiza estas cifras a fondo" assistant: "Usaré el agente de análisis para..." </example> model: inherit # inherit | sonnet | opus | haiku color: blue # blue|cyan|green|yellow|magenta|red tools: ["Read", "Bash", "Write"] # restringir herramientas --- # System Prompt del Agente (aquí va su "Soul") Eres un especialista en análisis de datos de marketing... ## Cómo comportarte - Solo trabaja con datos reales del usuario - Empieza siempre por el hallazgo más importante

El concepto de "Soul / Persona"

No existe un Souls.md oficial. El "alma" de un agente vive dentro del cuerpo del AGENT.md (es su system prompt). Define lo siguiente:

Rol

Quién es y desde qué perspectiva responde.

Valores

Qué puede y no puede hacer. Líneas rojas.

Tono

Cómo se comunica, qué formato usa, qué evita.

Proceso

Secuencia de pensamiento paso a paso.

Opciones de Modelo por Agente

Valor Uso Ideal
inherit Usa el mismo de la sesión. Por defecto.
opus Razonamiento profundo (más lento/caro).
sonnet Balance perfecto rendimiento/velocidad.
haiku Rápido y barato para clasificar/filtrar.

4 Hooks — Automatización Basada en Eventos

Lo que en otros sistemas se llama "Workflows". Son acciones automáticas que se disparan solas (se definen en hooks/hooks.json dentro de un plugin). No existe un Workflows.md en Cowork.

Eventos Principales

SessionStart Cargar contexto al inicio de sesión.
UserPromptSubmit Enriquecer el mensaje antes de procesar.
PreToolUse Validar seguridad antes de ejecutar tool.
PostToolUse Loguear acciones de herramientas.
PreCompact Guardar info antes de que la memoria se comprima.
SessionEnd Guardar notas y actualizar memoria base.
Prompt-Based

Claude evalúa y decide (approve/block). Bueno para lógica compleja.

"type": "prompt"
Command-Based

Ejecuta script bash directo (determinista). Bueno para checks fijos.

"type": "command"
hooks.json
{ "PreToolUse": [{ "matcher": "Write|Edit", // Solo para escribir "hooks": [{ "type": "prompt", "prompt": "Verifica convención proyecto" }] }] }

5 Arquitectura: Formas de Ampliar Cowork

Plugin, Connector, Project y Scheduled Task son herramientas completamente diferentes para propósitos distintos.

Concepto Qué es Qué resuelve
Plugin Bundle instalable (Skills+Agents+Hooks+MCP). "Quiero que Claude sepa hacer X tipo de trabajo"
Connector (MCP) Conexión a app externa (Slack, HubSpot…). "Quiero que Claude acceda a mis herramientas"
Project / Workspace Carpeta local montada. Claude lee/escribe ahí. "Quiero que Claude trabaje en mis documentos locales"
Scheduled Task Prompt que corre por horario (cron). "Que Claude haga esto cada día automáticamente"
Memory (CLAUDE.md) Contexto persistente entre sesiones. "Que Claude recuerde quién soy a largo plazo"

Clave sobre Scheduled Tasks

Cada ejecución arranca como una sesión completamente nueva sin memoria de ejecuciones anteriores. El prompt debe ser totalmente autocontenido.

Bien: "Abre ventas.xlsx, analiza y envía email."
Mal: "Continúa con el análisis que hicimos ayer..." (No hay "ayer").

6 Contexto, Persistencia y Tokens

La ventana de contexto es la memoria activa de Claude en una sesión (su "pizarra"). Cuando se llena, activa la Compactación (Context Compaction) y resume historial perdiendo detalles.

Peso en la Ventana de Contexto

Archivos leídos (Bajo demanda) ~50%
Historial de Mensajes (Acumulativo) ~40%
Resultados de Tools (Variables) ~30%
Skill Body (Solo si activo) ~25%
System Prompt (Fijo) ~20%
Memory / Metadata (Fijo) ~10%

Por defecto: NO SE GUARDA

Al cerrar sesión, el contexto desaparece. La siguiente sesión empieza de cero. Tampoco se guarda el contexto propio de los subagentes (solo devuelven el resultado final al padre).

Lo que sí sobrevive

  • Memory (CLAUDE.md): Preferencias y estilo.
  • Workspace: Archivos físicos escritos en disco.
  • Artifacts: Páginas HTML generadas.

Estrategias de Eficiencia

  • Haiku para tareas simples: Usar modelo rápido y 10x más barato para clasificar.
  • Delegar a Subagentes: No contamina el chat principal con el proceso.
  • Leer archivos parciales: Usar offset/limit en archivos gigantes.
Arquitectura Avanzada

Progressive Disclosure

Uno de los conceptos más importantes para entender cómo escalar plugins sin que se disparen los costes ni se contamine el contexto.

El problema que resuelve

Imagina que tienes un plugin con 10 skills instalados. Si cada SKILL.md ocupa 2.000 palabras (~2.600 tokens), en cada mensaje del usuario Claude tendría que cargar 26.000 tokens extra solo para "tener los skills disponibles".

Con un modelo de alto rendimiento como Sonnet, eso son céntimos por mensaje que se acumulan muy rápido. Además, llenas la ventana de contexto con información técnica que quizás no necesitas en esa sesión. La Progressive Disclosure soluciona esto con tres capas de carga.

Diseño Ingenuo 30.000 Tokens cargados SIEMPRE

Las 3 Capas en Detalle

Capa 1 — Metadata

Siempre en contexto (Coste mínimo)

Solo el name y la description del frontmatter YAML. Son los ~100 tokens que Claude necesita para decidir si el skill aplica. Nada más.

--- name: campaign-analyzer description: "Analiza archivos..." ---

10 skills × 100t = 1.000 tokens

Capa 2 — Body

Solo cuando se activa

El cuerpo completo del SKILL.md. Solo se carga en el contexto cuando Claude decide que ese skill específico aplica a la petición actual del usuario.

10 skills × 100t (meta) = 1.000t + 1 skill activo × 3.000t = 3.000t Total en contexto = 4.000t

Frente a los 30.000 del diseño ingenuo.

Capa 3 — References

Bajo demanda explícita

Los archivos en references/ nunca entran automáticamente. Claude usa la herramienta Read para leerlos solo si el SKILL.md se lo indica.

campaign-analyzer/ ├── SKILL.md └── references/ ├── roas-avanz.md ← bajo demanda └── casos-edge.md ← bajo demanda

Puede que nunca se carguen en el 90% de los casos.

Cómo diseñar un skill correctamente

La regla práctica es: pon en el body solo lo que se necesita en el caso más común. Todo lo demás va a references/. El truco está en que el SKILL.md diga explícitamente cuándo leer cada referencia:

## Análisis estándar [instrucciones para el 90% de casos] ## Casos avanzados (ROAS multi-touch, atribución) Si el usuario menciona atribución o multi-touch, lee references/roas-avanzado.md antes de continuar.

Mal Diseño

Todo en SKILL.md (3.800 tokens siempre)

SKILL.md (3.800 tokens)
├── instrucciones básicas
├── casos edge poco frecuentes
├── guía completa de fórmulas Excel
├── lista de todos los benchmarks del sector
└── ejemplos detallados de cada output posible

Buen Diseño

Body Lean + References (800 tokens)

SKILL.md (800 tokens)
├── instrucciones para el 90% de casos
└── "Lee references/X si el usuario pide Y"

references/
├── formulas-excel.md ← solo si genera Excel avanzado
├── benchmarks-sector.md ← solo si pide comparativa
└── casos-edge.md ← solo si hay datos atípicos

El efecto compuesto

Hay un efecto compuesto que mucha gente no ve: en una conversación larga, el body del skill activado permanece en el contexto durante todos los mensajes siguientes de esa sesión. Si el skill body es grande, contamina toda la conversación.

Con references bien usadas, Claude carga una referencia, la usa, y aunque permanezca en el historial, al menos solo está ahí porque fue necesaria, no porque estaba siempre.

La otra gran ventaja es el mantenimiento: cuando el skill tiene 10 archivos en references/ bien nombrados, es fácil actualizar. Si cambian los benchmarks del sector, solo tocas benchmarks-sector.md sin abrir ni romper el SKILL.md principal.

Resumen Mental: La Biblioteca

  • La ficha de catálogo (metadata) siempre está disponible, ocupa nada.

  • El índice y primer capítulo (body) los lees cuando el título te interesa.

  • Los apéndices técnicos (references) los consultas solo si necesitas ese detalle concreto.

Tú controlas exactamente qué va en cada capa. No es automático — es una decisión de diseño que tiene un impacto real en coste y rendimiento.

Estrategia Avanzada

Determinismo vs Instrucciones en Texto

Esta es una de las estrategias más potentes y menos obvias del sistema. Consiste en delegar la lógica algorítmica y repetitiva a scripts deterministas externos en lugar de forzar a Claude a interpretarla y programarla en cada ejecución.

El Problema con Explicar Procesos en Texto

Cuando intentas describir un flujo algorítmico (por ejemplo, cómo estructurar, calcular un CPA y ordenar un DataFrame Excel) mediante instrucciones en el cuerpo del SKILL.md:

## Cómo generar el ranking de campañas
1. Lee el archivo Excel con pandas
2. Detecta si existen las columnas 'spend' y 'conversions'
3. Si existen, calcula el CPA dividiendo spend entre conversions
4. Si conversions es 0, pon NaN para evitar división por cero
5. Ordena el dataframe por CPA ascendente...

Impacto negativo:

  • Consumo de tokens: Claude "piensa" en cómo implementar en vez de ejecutar.
  • Variabilidad: Distintos nombres, estilos y manejo de errores en cada chat.
  • Bugs invisibles: Imposible depurar un código que cambia dinámicamente.

La Alternativa: El Script Ya Escrito

En vez de intentar describir la lógica para que Claude la construya, almacenas el script completo directamente en la carpeta scripts/ y le ordenas a Claude que simplemente lo ejecute:

## Generar el ranking de campañas
Ejecuta el script de análisis:
```bash
python scripts/analizar_campañas.py --input "$ARCHIVO" --output "$SALIDA"
```
El script devuelve un JSON con el ranking y los semáforos. Interpreta y resume.

Impacto positivo:

  • Código robusto: Escrito, testeado previamente y con logs de error.
  • Rapidez: Ejecución instantánea sin tokens de razonamiento extra.
  • Determinismo: Siempre devolverá exactamente la misma estructura de datos.

Comparativa de Flujos Operativos

Flujo Tradicional (Instrucciones)
Petición del usuario
Claude lee las instrucciones (Tokens)
Claude razona cómo codificarlas
Claude genera el código en vivo
Claude ejecuta el script generado
Interpretación del resultado
Flujo Optimizado (Scripts Deterministas)
Petición del usuario
Claude lee una línea de ejecución rápida
Claude ejecuta el script directamente
Interpretación del resultado
¡Elimina por completo el razonamiento e implementación mecánica de código en vivo!

Cuándo usar Scripts vs Instrucciones

La regla es simple: si el proceso es algorítmico y determinista, debe ser un script. Si requiere criterio, juicio o adaptación estratégica, se definirá mediante instrucciones en texto en el SKILL.md.

Tarea / Proceso Script ✅ (Mecánico) Instrucciones ✅ (Criterio)
Calcular ROAS, CPA, CTR de un Excel
Recalcular fórmulas de hojas de cálculo
Detectar anomalías estadísticas mecánicamente
Interpretar qué significan esas anomalías para el negocio
Generar colores de un semáforo según umbrales fijos
Redactar recomendaciones del informe final
Validar mecánicamente si existen ciertas columnas
Decidir qué campañas de marketing pausar
Patrón Híbrido Recomendado: El script realiza el trabajo mecánico y determinista, y el cuerpo del SKILL.md guía el criterio cognitivo e interpretación.

Ejemplo real: El Skill de Excel (xlsx)

El skill de Excel que tienes instalado utiliza exactamente esta estrategia mediante un script llamado scripts/recalc.py. En vez de que Claude intente adivinar el cálculo de fórmulas Excel internamente (lo cual es impreciso y lento), el SKILL.md simplemente ordena ejecutar el script usando LibreOffice por debajo:

# Lógica dentro del SKILL.md ## Recalcular fórmulas (OBLIGATORIO si usas fórmulas)
```bash
python scripts/recalc.py output.xlsx
```

El script devuelve un JSON con los errores detectados:
```json
{"status": "errors_found", "total_errors": 2,
 "error_summary": {"#REF!": {"locations": ["Sheet1!B5"]}}}
```

Determinismo Absoluto

Con instrucciones en texto, ejecutar el mismo prompt 10 veces con el mismo archivo Excel puede generar 10 outputs distintos con lógicas cambiantes (diferentes formas de manejar NaN, uso de cut vs qcut).

Con un script determinista, 10 ejecuciones siempre devolverán 10 resultados exactamente idénticos. Esto es clave en un entorno corporativo donde el equipo que audita o lee el informe del lunes necesita recibir siempre la misma estructura de datos exacta.

Cómo Referenciar el Script desde el SKILL.md

Forma Directa (Script Autocontenido) Ejecuta el análisis completo:
```bash
python ${CLAUDE_PLUGIN_ROOT}/scripts/analizar.py \
  --input "$ARCHIVO" \
  --output "analisis_resultado.xlsx"
```
Forma Condicional (Script Opcional) ## Validación de columnas (siempre)
Lee el archivo y verifica que existen las columnas requeridas.

## Análisis estadístico (si es correcto)
Si la validación es correcta, ejecuta el análisis:
```bash
python scripts/stats_marketing.py --file "$ARCHIVO"
```

## Interpretación de negocio (siempre)
Independientemente del resultado, explica qué significan.

Importancia de ${CLAUDE_PLUGIN_ROOT}

Utiliza siempre esta variable de entorno para referenciar scripts internos dentro de tus plugins. Nunca uses rutas absolutas. Al usar esta variable, cuando el plugin se instale en otro ordenador o en un servidor de producción de un tercero, el sistema sabrá localizar dinámicamente y ejecutar el script sin romper las rutas de importación.
